22. The Project File#

22.1. Introduction#

To help organize your work and to create a clear, reproducable and explicit workflow, spaCy offers a feature called ‚Äúprojects.‚Äù A ‚Äúproject‚Äù is a file that acts like a template that runs various steps in your experiment. It‚Äôs a way to write everything down and clearly establish each step in your model training workflow. In this section, we‚Äôll cover the syntax of the project file and its main parts.

More here: spaCy projects docs

22.2. Syntax#

The project.yml file follows a format called yaml (‚Äúyet another markup language‚Äù), which is a simplified and human-friendly form for common data types such as Python lists and dictionaries. If you‚Äôve ever worked with markdown, this should feel somewhat familiar. Marksdown is a simplified format for HTML content. YAML is to data what markdown is to content.

The simplest way to write yaml is to separate a key and value with a semi-colon:

first-name: Paula
last-name: "Sloane"

which becomes a Python dictionary

{
  "first-name": "Paula", 
  "last-name": "Sloane"
}

Note that quotes are not required for string values, but can be used if you prefer.

To create a list, use the hyphen -.

- Fred
- Inna
- Oleh 

['Fred', 'Inna', 'Oleh']

These two basic forms can be combined to create complex objects. For example, let‚Äôs make up a dictionary of guests and what they‚Äôre bringing to a party. Note that indentation (two spaces) is used to create values that are themselves dictionaries or lists (called nesting).

guests:
  - name: Inna
    bringing:
      - tacos
      - ice cream
      - napkins
  - name: Oleh
    bringing:
      - pickles
      - hoagies  

{
    "guests": [
        {"name": "Inna", "bringing": ["tacos", "ice cream", "napkins"]},
        {"name": "Oleh", "bringing": ["pickles", "hoagies"]},
    ]
}

I find it helpful to remember that we‚Äôre basically building a JSON object. When in doubt, you can fall back on that syntax. Here‚Äôs a great app to test how your yaml will be converted into data: https://yaml-online-parser.appspot.com/

There‚Äôs much more that you can do with yaml, but let‚Äôs keep it simple for now. We‚Äôve covered the basics that you‚Äôll need for spaCy project files.

For more, here‚Äôs the awesome yaml page

22.3. Project file sections#

Most project files begin with basic metadata about the project. This is often just a title and description, but it can include whatever details you feel are important to give give context for the project. For example:

title: "New Language Model using UD Data"
authors: 
  - Sudeepta
  - Emma
description: "This project loads data from the Abyssinian UD treebank and trains a language model with a tagger and parser."
date: 29/12/2021

The variables section lets you clearly declare project settings and variables. These variables can be called and reused within the project using the format ${vars.your-variable}.

vars:
  config: "config.cfg"
  name: "ner_fashion"
  version: "0.0.0"
  # Change this variable if you want to use the GPU (gpu_id: 0)
  gpu_id: -1

The directories list is a list of project subfolders. If you want spaCy to create a new folder for you, just add it to the list.

directories: ["assets", "assets/conllu", "training", "configs", "corpus", "packages"]

The assets section describes the actions to be taken when you call spacy project assets. In most cases, this will fetch data from a remote source such as GitHub and download it to the computer currently running the project. This action ensures that anyone running the experiment will be using the same data and not something that‚Äôs specific to their computer.

For the example below, let‚Äôs say that vars.treebank is Yoruba. The code below will clone the New-Languages-for-NLP/Yoruba github repository and save all the files in the assets/Yoruba folder.

assets:
  - dest: "assets/${vars.treebank}"
    git:
      repo: "https://github.com/New-Languages-for-NLP/${vars.treebank}"
      branch: "main"
      path: ""

For more, see the spaCy docs

The workflows section makes it possible to run several commands in sequence. The most common use of this feature is spacy project run all. For example,

workflows:
  all:
    - install
    - convert

will run two commands: first, the install command, then the convert command. Commands can also be run individually, but the workflow provides a clear sequence in which they are meant to be run.

Probably the most important section of a project.yml file is the commands section. This section describes an action or step within the larger workflow. Each action has a name. It can also require certain inputs and outputs. The deps section will test that required files are present. An outputs section will confirm that the expected output files were created. The script section works like the command line. You can run several commands in sequence. You can use variables from the variables section. For example, the code below will run a python script called split.py to create files needed for model training with the test_size, random_state and lang variables. If there is more than one action in a command, you can add as many as you need.

commands:
  - name: split
    help: "Split the data into train, validation, and test"
    script: 
      - "python scripts/split.py ${vars.test_size} ${vars.random_state} ${vars.lang}"
    deps:
      - "corpus/converted"
    outputs:
      - "corpus/train.spacy"
      - "corpus/dev.spacy"
      - "corpus/test.spacy"

There‚Äôs much more to spaCy projects, but this outline should give you the essentials to understand how the file is formatted and what it does.
